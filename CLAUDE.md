# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Interview-to-Deck: an internal tool for consultants running qualitative interviews during audits/studies. Pipeline: capture interviews вЂ” either structured against a trame (Mission в†’ Trame в†’ Theme в†’ Question) or, since incr.9, freeform with AI-structured turns of speech and no trame вЂ” в†’ cross-interview AI synthesis в†’ PowerPoint restitution deck generated from a corporate template. Codebase and docs are French; domain vocabulary (`trame`, `synthГЁse`, `verbatim`, `recommandation`, `axe`) is established product terminology вЂ” keep it as-is, don't translate/anglicize.

Progress tracking lives in `.roadmap/roadmap.json` (rendered via the `roadmap-keeper` skill). Deeper onboarding docs are in `docs/wiki/` (architecture, conventions, stack), also viewable as a standalone rendered page at `docs/wiki.html` вЂ” both can lag behind the actual code/git history вЂ” verify against `git log`/`git status` rather than trusting either as current truth.

## Architecture

**Request flow**: FastAPI route (`app/routers/*.py`) в†’ service function (`app/services/*.py`) в†’ Jinja2 template (`app/templates/`), server-rendered with HTMX 2.0 for partial updates (autosave fields, dynamic panels). No SPA/build step on the frontend.

**Data model** (`app/models.py`, SQLAlchemy 2.0 / SQLite): `Mission` owns at most one `Trame` (в†’ `Theme` в†’ `Question`, optional since incr.9 вЂ” see below) and many `Interview` (в†’ `Answer`/`Verbatim` for a `mode="parametre"` interview that follows the trame). A mission also has one `GlobalSynthesis` (5 fixed categories: contexte, culture_adn, forces_succes, points_amelioration, aspirations) and several `RecommendationAxis` (в†’ `Recommendation`, scored 1-5 on valeur/complexitГ©). The PPT export and the web "aperГ§u" editor are two different renderings of the same `GlobalSynthesis`/`RecommendationAxis`/`Recommendation` data вЂ” there is no separate "slide" entity.

**Unified entry point + libre interviews** (incr.9, `app/routers/entretiens.py`): `/` (not `/missions`) is now the landing page вЂ” 3 choices (entretien libre, entretien structurГ©, nouvelle mission). The first two create a `Mission.is_draft=True` row immediately (an `Interview`/`Trame` always needs a `mission_id`) with a placeholder name; its real identity is completed afterward via `/missions/{id}/finaliser` (`missions.py`) вЂ” either naming it, or reattaching its interviews/trame to an existing non-draft mission (only offered when that mission has no trame yet, since `Mission`в†’`Trame` stays 1:1; reattachment reparents via a raw `sqlalchemy.update()` + `db.expire_all()` before deleting the now-childless draft, to sidestep the ORM's `delete-orphan` cascade on a re-parented one-to-one). `Interview.mode` (`parametre`/`libre`, set at creation, never exposed on any update route вЂ” the "lock" is structural, not a runtime check) distinguishes the two: `libre` has no `Trame` at all, instead `InterviewTurn` rows (interlocuteur/question/remarque, plus an optional `section_title` marking the turn that opens a new topic вЂ” inherited by following turns, see `interview_export.py::group_turns_into_sections()`, shared between the Analyse screen and the per-interview Markdown export below) produced by `interview_libre_extract_ai.extract_libre_from_text()` вЂ” one AI call, on the raw transcript before human review, that structures the turns into sections, drafts a rГ©partition into `GlobalSynthesis`'s 5 categories (stored as `Interview.repartition` JSON, same shape as `synthese_ai.GLOBAL_SCHEMA`), a 1-3 sentence `Interview.resume`, and detected identity вЂ” all reviewed together on one screen before saving. One read-only screen renders this as a document rather than a raw form (styled like the pre-existing `interviews/preview.html`, `.preview-doc` CSS): `GET /interviews/{id}/analyse` вЂ” turns grouped by section, then rГ©sumГ© + the 5 categories in prose, all on one page (built 2026-07-15 against two real example documents as a quality bar; merged from two separate screens into one on 2026-07-17 вЂ” see "Convergence libre/trame" below). `synthese_ai.generate_global_synthesis()`/`_build_global_prompt()` take an additional `material_libre` parameter (list of `(Interview, repartition)`, gathered by `synthese.py::_libre_material()`) injected as its own prompt section alongside `material_by_theme` вЂ” so a mission's global synthesis blends both interview types. Every template/route that assumed `mission.trame` always exists now guards for `None` (see "Mission sans trame" below for the 2026-07-17 fix that made this actually work end-to-end, not just avoid crashing). The saved audio backup (`Interview.audio_backup_path`, `data/recordings/`) is now actually served вЂ” `GET /missions/{id}/interviews/record/backup/{filename}` (path-traversal-guarded) вЂ” and linked from both record screens (appears once the backup upload completes) and `libre_detail.html`; before this it was written to disk but never exposed anywhere, a real gap a user caught in testing, not a regression in this repo's git history despite how it was reported. `audio_transcribe.py`'s Whisper defaults were raised `small`в†’`medium` + `beam_size` 1в†’2 (`WHISPER_BEAM_SIZE`, new) after a real-audio comparison (`tests/exemple/`, a ~114min recording with a human-edited reference transcript) showed `small`/greedy badly mangling proper nouns ("Michel Nakache"в†’"Michel Lacache", "Aurore"в†’"l'horror") вЂ” `medium`/beam=2 got them right; timing was inconclusive in the sandboxed dev environment, worth re-checking live-segment speed on real hardware. `ai_common.py` gained an `ollama` provider (plain `urllib` HTTP to a local server, no new dependency, no API key вЂ” `is_configured()`/`call_ai_json()` special-case it to skip the key check every other provider requires) so `interview_libre_extract_ai.py`'s analysis can run fully local, pairing with the already-local Whisper transcription; requires Ollama installed + a model pulled on the machine (not something this codebase can provision) вЂ” installed and tested end-to-end on the dev machine since 2026-07-17 with real audio (`llama3.1:8b`, `qwen2.5:3b-instruct`, `qwen2.5:1.5b-instruct` pulled). CPU-only inference is slow and is the confirmed bottleneck, not Whisper (`llama3.1:8b` ~55-70s for even a short paragraph; `qwen2.5:1.5b-instruct` ~4x faster, extraction quality trade-off unconfirmed) вЂ” `OLLAMA_TIMEOUT` and `OLLAMA_CHUNK_MAX_WORDS` (map-reduce chunk size, `interview_libre_extract_ai.py`) are both env-var-overridable (`.env.example`) for tuning per machine; a real timeout on a ~37min real interview was observed and is the main open perf question (see `docs/wiki.html`'s TODO table). Per-interview Markdown export (US9.7, `app/services/interview_export.py::build_interview_markdown()`, 2026-07-16): a "TГ©lГ©charger (.md)" button on `libre_analyse.html` and `preview.html` downloads `GET /interviews/{id}/export/markdown` вЂ” one interview's own content (libre: rГ©sumГ© + structured transcript by section + the 5-category rГ©partition; parametre: theme/question/answer/verbatim, like `preview.html`). Distinct from the pre-existing mission-wide `export.py::export_interviews`/`mission_export.build_export_markdown` (all interviews of a mission, theme-grouped, feeds the external-analysis-and-reimport circuit via `analyse_import.py`) вЂ” this one has no analysis-request template footer and isn't reimportable, it's a plain read/archive export. The independent UX/UI pass (US9.12, 4 parallel design-review agents, 2026-07-16) fed a round of concrete fixes through 2026-07-17 (wizard, autosave feedback, Q&A redesign, interview/mission synthesis decoupling, screen convergence below) вЂ” see `.roadmap/roadmap.json` increment 9 for the full list.

**Mission sans trame вЂ” synthГЁse/recommandations/export PPT (fix 2026-07-17)**: a mission with only libre interviews (`mission.trame is None`) used to 500 on `GET/POST /missions/{id}/recommandations` (`synthese.py`'s `recommendations_view`/`generate_recommendations_view` did `mission.trame.themes` unguarded) and, even once that's fixed, the 4 wizard screens (`synthese/recommandations.html`, `globale.html`, `apercu.html`, `export_import.html`) all gated their *entire* content вЂ” including the AI generation panels вЂ” behind `{% if not themes %}`, so a libre-only mission saw "la trame est vide" instead of its real synthesis even though `_all_theme_material`/`_libre_material` already fed libre material into it correctly. Both fixed: the guard is now `{% if not mission.interviews %}` (material presence, not trame presence). The legacy per-theme synthesis screen (`synthese_view`, `synthese/theme.html`/`_panel.html`, superseded since incr.9 by the Analyseв†’SynthГЁse globaleв†’Recommandationsв†’Export PPT wizard) was confirmed orphaned and removed rather than fixed.

**Convergence libre/trame вЂ” Г©cran "aperГ§u" (2026-07-17)**: a trame interview has exactly 2 screens (edit `capture.html`, read-only `preview.html`); a libre interview had the same conceptual content spread over 3 (edit `libre_detail.html`, read-only turns `libre_analyse.html`, read-only rГ©sumГ©/rГ©partition `libre_synthese.html`). Merged the two read-only screens into one (`libre_analyse.html` now renders turns-by-section then rГ©sumГ©/rГ©partition, matching `preview.html`'s single-page shape) вЂ” `GET /interviews/{id}/analyse/synthese` is now a 308 redirect to `/analyse` for link compatibility, `libre_synthese.html` is gone. `libre_detail.html`'s button relabeled "рџ‘ЃпёЏ AperГ§u" to match `preview.html`'s convention.

**AI provider abstraction** (`app/services/ai_common.py`): all AI calls (theme/global synthesis, recommendations, trame/interview extraction) go through one `call_ai_json(system, prompt, schema, json_hint, error_cls=...)` function. The active provider is chosen by `AI_PROVIDER` (**ollama**/openai/mistral, default ollama since 2026-07-17 вЂ” Anthropic support was removed entirely, no SDK dependency, no code path, at the user's request after hitting a billing/credit error) with no automatic fallback between providers вЂ” a missing key raises a clear error naming the exact env var expected (`api_key_env_name()`). `synthese_ai.py`, `trame_extract_ai.py`, `interview_extract_ai.py` each define their own system prompt/schema/error class but never touch an SDK directly. `SYNTHESE_DEMO=1` enables a rule-based offline fallback (no AI, no cost) used when no key is configured. `ollama` (incr.9, no new dependency вЂ” plain `urllib` HTTP calls to a local Ollama server, `OLLAMA_HOST` optional, defaults to `http://localhost:11434`) runs `interview_libre_extract_ai.py`'s analysis (turns/rГ©partition/identity) without sending interview content to an external AI вЂ” pairs with the already-local Whisper transcription for a fully offline "entretien libre" pipeline. Requires Ollama installed and a model pulled on the machine; now installed on the main dev machine (`qwen2.5:1.5b-instruct`, `qwen2.5:3b-instruct`, `llama3.1:latest` pulled, 2026-07-17) вЂ” `SYNTHESE_MODEL` defaults to `llama3.1` for this provider, quality on 7-8B local models vs. hosted-frontier models for this kind of structured JSON extraction is still an open question worth watching in practice.

**PPT export** (`app/services/pptx_export.py` + `app/services/pptx_deck.py`): `build_presentation()` either starts from a client-uploaded `.pptx` template (inherits theme/masters/logo вЂ” `_clear_slides()` must drop each removed slide's relationship via `prs.part.drop_rel()`, not just clear `sldIdLst`, or PowerPoint silently refuses to open the file even though python-pptx parses it fine) or builds a blank 16:9 deck (in which case the native title placeholder, inherited from python-pptx's 4:3 default, is repositioned explicitly per-slide in `_new_slide()` rather than by mutating layout/master placeholders, which is unreliable). Every slide-building function pulls sizing from `pptx_deck.D.TYPE` (the one type scale) and adapts font size to content length via `D.ajuster_police`/`D.estimer_lignes`/`D.tronquer_a_lignes` rather than assuming fixed line counts. `build_presentation()` always ends with `D.verifier_geometrie()`, raising if any shape ends up off-slide. A `.pptx` that parses in python-pptx and passes this check can still fail to open in real PowerPoint or look wrong visually вЂ” always verify a real export with the `pptx-verify` skill (render + eyeball) before considering a layout change done. See the `pptx-deck` and `restitution-deck-design` skills for the design system this generator follows.

**Web editor / PPT preview parity**: `app/templates/synthese/apercu.html` is a tabbed editor (one tab per slide) whose form fields reuse the same autosave endpoints as the rest of the app (`/syntheses/globale/{mission_id}/field`, `/recommandations/{id}/field`, `/recommandations/axes/{id}/field`). The "shape-aware" hint shown under each field comes from `pptx_export.field_fit_hint()`, which reuses the exact geometry constants and font-fitting functions the real PPT generator uses вЂ” treat that coupling as intentional; if you change a slide's layout constants in `pptx_export.py`, update `FIELD_SHAPE` too or the hint will drift from reality.

**New (2026-07-15) вЂ” `pptx-framed-image` and `slide-text-polish` skills**, grafted from a sibling project's PPT toolkit (`.claude/skills/`, tests re-run after copy: 9/9 and 9/9) вЂ” available but **not yet wired into `pptx_export.py`**, nothing in the generator changed. `pptx-framed-image` clips an image to a template frame's exact preset shape (`prstGeom` cloning) instead of a plain rectangle, and ships `stock_images.py` вЂ” a real royalty-free photo fetcher (Openverse, CC0, no API key) with a procedural offline fallback (`nature_images.py`); relevant if a client-uploaded template (`build_presentation()`'s template path) turns out to carry its own photo-frame placeholders worth filling properly rather than leaving blank or stretching an image into. `slide-text-polish` lints slide copy (title-as-claim, bullet length, filler, abbreviations) вЂ” usable on any `{title, bullets}` structure regardless of the export path. `docs/vscode1-export/` is now a full local mirror of the sibling project VSCode1's `export/` folder (`ppt-toolkit.md`, `points-amelioration-ppt.md`, `design-system-octo.md`, `template-octo.md`, `optimisation-tokens.md`, 2026-07-15) вЂ” same convention as the sibling project VSCode3 (which mirrors the same 5 files). Reference copy, not source of truth вЂ” `template-octo.md`/`design-system-octo.md` describe VSCode1's own OCTO template/pipeline, useful here as design-principle inspiration since this project's PPT export also targets an OCTO-branded template, not as literal paths. Re-sync manually if the VSCode1 originals evolve.

**OpenHub agents integration** (`app/services/openhub_agents.py`) is unrelated to Claude Code's own skill/agent system вЂ” it shells out to an external `opencode` CLI (`.opencode/agents/`, `.opencode/skills/`) to power the app's own "Agents" page, with a simulated-response fallback when `opencode` isn't on PATH.

## Discipline de gestion des tokens (2026-07-16, cf. `docs/vscode1-export/optimisation-tokens.md`)

Le contexte est un cache actif facturГ© Г  chaque tour, pas une mГ©moire gratuite вЂ” le laisser croГ®tre sans discipline pГ©nalise coГ»t, qualitГ© (В« lost in the middle В») et latence (source : OCTO Playbook Agentique, partie В« Optimiser la consommation Tokens В»). RГЁgles concrГЁtes, pas de changement de ton/style de rГ©ponse :

- **Ne pas parcourir** `_bmad/`, `_bmad-output/`, `.venv/`, `__pycache__/`, `.git/`, `data/`, `.pytest_cache/` sauf demande explicite вЂ” hors du pГ©rimГЁtre utile Г  99 % des tГўches.
- **Lire avant d'Г©crire, grep les appelants avant de modifier** une fonction/un champ partagГ© (dГ©jГ  la pratique de ce projet вЂ” l'expliciter la rend systГ©matique).
- **PrГ©fГ©rer un grep/read ciblГ© Г  un dump rГ©cursif** вЂ” surtout sur `.claude/skills/bmad-*` (~46 skills) qui gonfle vite le contexte si parcouru en entier.
- **Sous-agent pour toute sortie volumineuse** (exploration large, longs logs) plutГґt que de la laisser polluer le contexte principal.
- **`/compact` dГЁs ~40 %** de fenГЄtre de contexte utilisГ©e si la conversation doit continuer longtemps sur le mГЄme sujet.
